1. Understanding Universal SDP IDs

What is a Universal SDP ID (USDP)?

A Universal SDP ID is an Ontario-wide unique 8-digit identifier assigned by the MDM/R that identifies a Service Delivery Point (SDP) across all LDCs and market participants in the province.

Key Characteristics

  • Format: Fixed 8-digit number (e.g., 00037482)
  • Uniqueness: Unique across all of Ontario, not just within your LDC
  • Assignment: Can only be assigned by MDM/R through File Type 1000/2000 process
  • Permanence: Once assigned, the USDP ID remains with that SDP for its lifetime
  • Prerequisite: Must be assigned BEFORE you can create the SDP in MDM/R through synchronization

USDP ID vs SDP ID

Identifier Type Universal SDP ID (USDP) SDP ID
Purpose Ontario-wide unique identifier for province-wide market operations Your LDC's internal identifier for the service delivery point
Assigned By MDM/R (via File Type 1000/2000 process) Your LDC's systems
Format Fixed Number(8), zero-padded (e.g., 00037482) Varchar(50), any format your LDC uses
Uniqueness Unique across all Ontario LDCs Unique only within your LDC
Used In All MDM/R transactions, market settlements, retailer operations Your internal systems and synchronization file sets
Relationship ONE SDP ID maps to ONE USDP ID. This mapping is established during USDP assignment and maintained throughout the SDP's lifetime.

2. When to Request USDP IDs

Critical Timing: Request BEFORE Synchronization

‚ö†Ô∏è CRITICAL PREREQUISITE

You MUST obtain a Universal SDP ID BEFORE you can create a new SDP in the MDM/R through synchronization. If you attempt to synchronize a new SDP without first obtaining its USDP ID, the synchronization will fail with error code 10 in the IR14 report: "Universal SDP ID not defined through Universal SDP ID Assignment Process."

Request USDP IDs When:

  • New Construction: New homes, buildings, or facilities are being connected to your distribution system
  • Service Upgrades: Existing location is being upgraded to create a new metering point
  • Subdivisions: Multiple new SDPs in a new development
  • Service Splits: One SDP is being divided into multiple separate SDPs
  • Historical Cleanup: Identifying SDPs that exist in your system but were never properly registered in MDM/R
  • Bulk Requests: You can request thousands of USDP IDs in a single file for planned projects

Recommended Workflow

1
Identify New SDPs in Your System
Determine which locations need Universal SDP IDs (new construction, upgrades, etc.)
2
Create File Type 1000 Request
Submit request file with your LDC_ID and internal SDP_ID for each location
3
MDM/R Processes Request (15-min cycles)
Receive File Type 2000 Response and IR01 Report
4
Map USDP IDs to Your SDP IDs
Update your internal systems with the assigned Universal SDP IDs
5
Create SDPs via Synchronization
Now you can synchronize these SDPs to MDM/R using File Type 4000 with both SDP_ID and Universal_SDP_ID

3. Prerequisites & Requirements

Before You Can Request USDP IDs

Requirement Details
MDM/R Enrollment Your LDC must be enrolled with MDM/R and assigned an Organization ID (ORG_ID) in format ORGxxxxx
AS2 Connection File Transfer Services (FTS) must be configured with AS2 connectivity to MDM/R. Reference SME_MAN_9001 for AS2 setup.
Agent Authorization If using an agent to submit requests on your behalf, the agent relationship must be registered with MDM/R
Internal SDP IDs You must have internal SDP identifiers in your systems. These can be any format (up to 50 characters) but must be unique within your LDC.
LDC ID Format Must be exactly 8 characters: Char(8) format, example: ORG12345

Required Information for Each Request

  • LDC_ID: Your organization identifier (Char 8)
  • SDP_ID: Your internal identifier for the SDP (Varchar 50, unique within your LDC)

Important Notes

  • You can request multiple USDP IDs in a single file (hundreds or thousands)
  • If you request a USDP ID for an SDP_ID that already has one, it will be flagged as "already issued" (not an error)
  • Invalid LDC_ID will cause request to fail - verify your ORG_ID is correct

4. The USDP Assignment Process Flow

Overview of the Request/Response Cycle

1
LDC Submits File Type 1000
Universal SDP ID Assignment Request via FTS (AS2)
2
FTS Processing
File passes AS2, undergoes FTS validation (file naming, authorization)
Failure = IR17 Report
3
MDM/R Processes Request
Validates LDC_ID, checks for duplicates, assigns new USDP IDs
Processes every 15 minutes from 00:00 to 23:45 EST
4
MDM/R Generates Outputs
Creates File Type 2000 Response + IR01 Summary Report
5
LDC Receives Files
File Type 2000 with assigned USDP IDs + IR01 report with summary statistics via FTS and GUI

Processing Schedule & Timing

15-Minute Processing Windows

The USDP Assignment process runs as a scheduled script every 15 minutes from 00:00:00 EST to 23:45:00 EST daily.

  • Processing Windows: :00, :15, :30, :45 of each hour
  • Last Run: 23:45 EST - no processing from 23:45 to midnight
  • Batching: Multiple requests submitted within the same 15-minute window are processed together
  • Report Generation: One IR01 report is generated per 15-minute cycle, containing summary records for all requests processed in that window

Timing Considerations

  • Late Submission: If your request completes processing after midnight (or the day after submission), you may receive an empty IR01 report with message: "There are no POD Id requests to report for the given condition"
  • Best Practice: Submit requests earlier in the day to ensure same-day processing and reporting
  • Expect Response: Within 15 minutes during business hours, you should receive File Type 2000 and IR01

File Flow Summary

File Type Direction Purpose Delivery Method
1000 LDC ‚Üí MDM/R Request assignment of Universal SDP IDs FTS (AS2)
2000 MDM/R ‚Üí LDC Response containing assigned USDP IDs FTS (AS2)
IR01 MDM/R ‚Üí LDC Summary exception report with statistics FTS (AS2) + GUI
IR17 MDM/R ‚Üí LDC FTS processing failure (if FTS errors occur) FTS (AS2) only

5. Creating the Request File (File Type 1000)

File Naming Convention

File Type 1000 follows the standard MDM/R file naming convention:

<ORG_ID_1>.<ORG_ID_2>.1000.<REQ_VER>.<DATE_TIME>.DAT
Element Format Description
ORG_ID_1 Char(8) The LDC Organization ID (e.g., ORG11111)
ORG_ID_2 Char(8) The submitting organization ID (LDC or authorized agent, e.g., ORG22222)
1000 Fixed File Type identifier for USDP Assignment Request
REQ_VER Number(2) Request version (currently 00)
DATE_TIME yyyyMMddHHmmss Date and time file was created (EST)
.DAT Fixed File extension

File Naming Examples

Example 1 - LDC submits directly: ORG11111.ORG11111.1000.00.20250116120000.DAT Example 2 - Agent submits on behalf of LDC: ORG11111.ORG22222.1000.00.20250116120000.DAT

File Structure & Format

File Type 1000 is a pipe-delimited (|) text file. Refer to the MDM/R Universal SDP ID Assignment Technical Interface Specification (SME-SPEC-39) for complete file format details including:

  • File Name Record: Embedded file name in <FTSFN> tags
  • Header Record: Contains LDC_ID and other metadata
  • Detail Records: One record per SDP_ID for which you're requesting a USDP ID

Key Points for Request File Creation

  • File must be pipe (|) delimited
  • All dates/times must be in EST (never EDT)
  • LDC_ID must be exactly 8 characters (Char 8)
  • SDP_ID can be up to 50 characters (Varchar 50)
  • Each SDP_ID must be unique within your LDC
  • You can include hundreds or thousands of requests in one file
  • File must not use pipe character (|) within data fields

Required Data Elements

For each SDP you're requesting a USDP ID for, you must provide:

  1. LDC_ID: Your organization identifier (must match enrolled ORG_ID)
  2. SDP_ID: Your internal identifier for this service delivery point

Common Request File Mistakes

  • ‚ùå Using invalid/unregistered LDC_ID ‚Üí Will create "Number of invalid LDC IDs" exception
  • ‚ùå Incomplete data (missing required fields) ‚Üí Will create "Number of Incomplete Data Requests" exception
  • ‚ùå Requesting USDP ID for SDP_ID that already has one ‚Üí Will show as "Number of already issued Ids" (not an error, just informational)
  • ‚ùå Using EDT instead of EST for timestamps
  • ‚ùå Including pipe character (|) in SDP_ID or other data fields

6. Understanding the Response File (File Type 2000)

Response File Overview

File Type 2000 is generated by MDM/R and sent back to you via FTS. It contains the assigned Universal SDP IDs mapped to your SDP IDs.

File Naming Convention

Response files follow the same naming pattern as requests but with File Type 2000:

<ORG_ID_1>.<ORG_ID_2>.2000.<REQ_VER>.<DATE_TIME>.DAT

Response File Content

The response file maps your SDP_IDs to the newly assigned Universal SDP IDs:

Data Element Format Description
LDC_ID Char(8) Your organization identifier
SDP_ID Varchar(50) Your internal SDP identifier (from request)
Universal_SDP_ID Fixed Number(8) The newly assigned Universal SDP ID (e.g., 00037482)

Using the Response File

Action Required: You must update your internal systems to map each SDP_ID to its assigned Universal_SDP_ID. This mapping is permanent and must be maintained for the lifetime of the SDP.

Next Steps: Once you have the Universal SDP IDs, you can proceed to create these SDPs in MDM/R through the Incremental Synchronization process (File Type 4000).

Response File Not Created?

When Response File is NOT Generated

If your request had critical errors, File Type 2000 may not be created. In this case:

  • The IR01 report will have "Output File Name" field = null
  • The "Status" in IR01 will show ERROR
  • Check IR01 for specific error counts (invalid LDC IDs, incomplete data, etc.)
  • Correct the issues and resubmit File Type 1000

Parsing the Response

Refer to SME-SPEC-39 (MDM/R Universal SDP ID Assignment Technical Interface Specification) for complete details on:

  • Response file structure (header record, detail records)
  • Field formats and delimiters
  • Data types and lengths
  • Record sequencing

7. Interpreting the IR01 Report

Report Purpose

The Universal SDP ID Assignment Request Summary Exception Report (IR01) provides summary statistics for each File Type 1000 request processed, including success counts and error counts.

Report Delivery

Delivery Method Details
FTS (AS2) Sent via File Transfer Services to the submitting organization
GUI Available through the MDM/R Reports tab for LDC authorized users
File Name ORG_ID_1.ORG_ID_2.IR01.00.yyyyMMddHHmmss.DAT

IR01 Report Format

Header Information

  • Report File name: IR01 file name
  • Report Name: Universal SDP ID Assignment Request Summary Exception Report
  • Report Organization: LDC Organization ID
  • Report Date/Time: When report was generated (yyyy-MM-dd HH:mm:ss GMT-05:00)

Detail Record Fields

Field Name Format Description
Request Date yyyy-MM-dd HH:mm:ss Date and time the request was sent to MDM/R
Input File Name Varchar(250) Name of your File Type 1000 request file
Output File Name Varchar(250) Name of File Type 2000 response (null if response not created due to errors)
Status Varchar(7) SUCCESS or ERROR
LDC Id Char(8) Organization ID of the LDC
Number of requested Ids Number(8,0) Total number of USDP IDs requested in the file
Number of issued Ids Number(8,0) Total number of NEW USDP IDs assigned in the response
Number of already issued Ids Number(8,0) SDP_IDs that already had USDP IDs assigned (duplicates - informational)
Number of invalid LDC IDs Number(8,0) Exceptions due to invalid/unregistered LDC_ID
Number of Incomplete Data Requests Number(8,0) Exceptions due to missing required fields

Interpreting Status Values

Status: SUCCESS

Meaning: All records in the file were processed successfully.

What to Check:

  • Number of issued Ids = Number you expected
  • Number of already issued Ids = Expected duplicates (if any)
  • Output File Name has a valid File Type 2000 file name

Next Steps: Retrieve File Type 2000, parse the USDP IDs, update your internal systems

Status: ERROR

Meaning: The file was processed with errors.

What to Check:

  • Number of invalid LDC IDs > 0: LDC_ID is not recognized by MDM/R. Verify your ORG_ID.
  • Number of Incomplete Data Requests > 0: One or more records missing required data fields. Review file format against SME-SPEC-39.
  • Output File Name = null: Response file was NOT created due to errors.

Next Steps: Correct the errors identified and resubmit File Type 1000

Understanding "Already Issued" IDs

Not an Error - Informational

The "Number of already issued Ids" field counts SDP_IDs that already have Universal SDP IDs assigned. This is NOT an error - it simply means:

  • You previously requested a USDP ID for this SDP_ID
  • The mapping already exists in MDM/R
  • The original USDP ID assignment remains valid
  • This can happen if you resubmit a request or if there was confusion about which SDPs needed USDP IDs

Action: Verify in your systems whether you have the USDP ID for these SDP_IDs. If not, you may need to retrieve it from historical records or contact MDM/R Support.

Multiple Requests in One IR01

15-Minute Batching

If multiple File Type 1000 requests are processed within the same 15-minute window, the IR01 report will contain multiple detail records - one for each request file. Each detail record will have its own Input File Name, Status, and counts.

Empty IR01 Report

Timing Issue Message

If you receive a message: "There are no POD Id requests to report for the given condition"

Cause: Your request was submitted at a time when the USDP ID generation completed after midnight or the day after submission.

Resolution: Check for File Type 2000 response. If you have the response file, the assignment was successful despite the empty report. For future requests, submit earlier in the day.

8. Troubleshooting Common Issues

Problem: Received IR17 Instead of IR01

FTS Processing Failure

Cause: Your File Type 1000 failed FTS validation before reaching the USDP assignment process.

Common FTS Errors:

  • E120: File name syntax violation - verify file naming convention
  • E140: Not authorized to submit File Type 1000 - check enrollment status
  • E150: AS2 sender ID doesn't match ORG_ID_2 in file name

Resolution: Review IR17 troubleshooting section in the IR Reports guide for detailed FTS error resolutions.

Problem: IR01 Status = ERROR with Invalid LDC IDs

Symptom Resolution
Number of invalid LDC IDs > 0 Cause: LDC_ID in your request does not match any enrolled organization in MDM/R.
Check: Verify your ORG_ID is correct (should be ORGxxxxx format, exactly 8 characters).
Action: Correct the LDC_ID in your File Type 1000 and resubmit. If you believe your ORG_ID is correct, contact MDM/R Service Desk to verify enrollment status.

Problem: IR01 Status = ERROR with Incomplete Data

Symptom Resolution
Number of Incomplete Data Requests > 0 Cause: One or more detail records in your File Type 1000 are missing required data fields.
Common Issues:
  • Missing SDP_ID
  • Missing LDC_ID
  • Incorrect number of pipe-delimited fields
  • Empty/null values in required fields
Action: Review SME-SPEC-39 for complete File Type 1000 format. Ensure all required fields are present and properly delimited. Validate your file generation process and resubmit.

Problem: No Response File (Output File Name = null)

Cause: Critical errors prevented response file generation.

Check IR01 for:

  • Status = ERROR
  • Number of invalid LDC IDs > 0 (most common)
  • Number of Incomplete Data Requests > 0

Resolution: Fix the errors indicated in IR01 counts and resubmit File Type 1000.

Problem: All Requests Show "Already Issued"

Not a Problem - Already Have USDP IDs

Meaning: All SDP_IDs in your request already have Universal SDP IDs assigned.

This means:

  • You previously requested USDP IDs for these SDP_IDs
  • The USDP IDs are already established in MDM/R
  • You should have the USDP IDs in your internal systems (or in a previous File Type 2000 response)

Action: Check your historical records for the USDP ID assignments. If you cannot locate them, contact MDM/R Service Desk for assistance in retrieving the existing mappings.

Problem: Some Requests Issued, Some Already Issued

Partial Success - Normal Scenario

Meaning: You submitted a mix of new SDP_IDs (which received new USDP IDs) and SDP_IDs that already had USDP IDs.

This is normal when:

  • Doing bulk requests that include both new and existing SDPs
  • You're not certain which SDPs already have USDP IDs
  • Resubmitting a corrected file that had some successes initially

Action: File Type 2000 will include USDP IDs for all records (both newly issued and already issued). Update your systems with any new assignments.

Problem: Empty IR01 Report Received

Symptom Resolution
Message: "There are no POD Id requests to report for the given condition" Cause: Request processing completed after midnight or the day after submission.
Impact: USDP IDs may still have been assigned successfully.
Action:
  1. Check for File Type 2000 response file
  2. If File Type 2000 exists, assignments were successful
  3. If no File Type 2000, resubmit request earlier in the day

Problem: Cannot Use USDP ID in Synchronization

IR14 Error Code 10

Symptom: Synchronization fails with IR14 error code 10: "Universal SDP ID not defined through Universal SDP ID Assignment Process"

Cause: You're trying to create an SDP in MDM/R using a Universal SDP ID that was never assigned (or was assigned incorrectly).

Resolution:

  1. Verify the USDP ID format is correct: Fixed Number(8), zero-padded (e.g., 00037482, not 37482)
  2. Check that you received this USDP ID in a File Type 2000 response
  3. Verify IR01 status was SUCCESS for the request
  4. If USDP ID is from "already issued" category, verify it's the correct one
  5. If still failing, contact MDM/R Service Desk to verify USDP ID exists in system

Problem: Mismatch Between Request Count and Issued Count

Math Check

Formula: Number of requested Ids = Number of issued Ids + Number of already issued Ids + Number of invalid LDC IDs + Number of Incomplete Data Requests

If numbers don't add up:

  • Review your File Type 1000 structure
  • Ensure each detail record represents one SDP_ID request
  • Check for duplicate SDP_ID entries in your request file
  • Verify file format against SME-SPEC-39

9. Best Practices & Tips

Planning & Timing

  • Request Early: Submit USDP ID requests well before you need to synchronize the SDPs (days or weeks ahead for bulk projects)
  • Bulk Requests: For large projects (subdivisions, new developments), request all USDP IDs in one file to simplify tracking
  • Submit During Business Hours: Submit between 9 AM and 4 PM EST to ensure same-day processing and avoid midnight timing issues
  • Allow 15-30 Minutes: Expect File Type 2000 and IR01 within one or two 15-minute processing cycles

File Creation

  • Validate Before Submission: Check file format, field counts, and data types before submitting
  • Use Consistent Naming: Follow MDM/R file naming convention exactly (case-sensitive)
  • EST Always: All timestamps must be in EST, never EDT - even during daylight saving time
  • Avoid Special Characters: Do not use pipe (|) character in SDP_ID or other data fields
  • Document Requests: Keep records of which SDP_IDs were included in each File Type 1000 for troubleshooting

Data Management

  • Maintain Mapping: Store the SDP_ID to USDP ID mapping permanently in your database
  • Archive Response Files: Keep all File Type 2000 response files for historical reference and audit purposes
  • Track IR01 Reports: Monitor IR01 reports to identify trends in errors or duplicate requests
  • Unique SDP IDs: Ensure your internal SDP_IDs are truly unique within your LDC before requesting USDP IDs

Error Prevention

  • Verify ORG_ID: Confirm your LDC_ID (ORG_ID) is correct and enrolled before first submission
  • Test with Small Batch: For first-time submissions, test with a small batch (5-10 SDPs) before bulk requests
  • Check "Already Issued": Before requesting, verify your internal systems to avoid unnecessary "already issued" responses
  • Monitor FTS: Ensure AS2 connection is stable before submitting large batches

Integration Workflow

  • Automate Parsing: Build automated processes to parse File Type 2000 and update internal databases
  • Workflow Sequence: Build business processes that enforce: Request USDP ‚Üí Receive Response ‚Üí Update Systems ‚Üí Then Synchronize SDP
  • Exception Handling: Implement automated alerts for ERROR status in IR01
  • Reconciliation: Regularly reconcile your SDP_ID to USDP_ID mappings against MDM/R

Common Pitfalls to Avoid

‚ùå Don't Do These

  • Synchronize Before Getting USDP ID: Always get USDP ID first, then synchronize
  • Reuse USDP IDs: Never assign the same USDP ID to multiple SDPs in your system
  • Change SDP_ID After Assignment: Once mapped to a USDP ID, don't change the SDP_ID in your system
  • Request Without Verifying LDC_ID: Always confirm your ORG_ID before bulk requests
  • Ignore "Already Issued": These indicate existing USDP IDs - track them down rather than re-requesting
  • Submit at 11:50 PM: Avoid late submissions that may process after midnight

Getting Help

  • Documentation: Reference SME-SPEC-39 for complete file format specifications
  • Service Desk: Contact MDM/R Service Desk for enrollment issues, missing USDP IDs, or system problems
  • Test Environment: If available, use test environment for first-time file creation
  • Peer Review: For large bulk requests, have another team member review file before submission
