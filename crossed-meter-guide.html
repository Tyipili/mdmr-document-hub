1. Understanding Crossed Meters

1.1 What Are Crossed Meters?

Definition

Crossed meters (also known as mixed meters) occur when two or more meters have been reported as installed on the incorrect Service Delivery Point (SDP) in your Local Distribution Company's system.

The meters have therefore been associated with the wrong SDP when the SDP to Meter Relationship was synchronized with the MDM/R.

1.2 How Crossed Meters Happen

Common causes of crossed meter scenarios:

  • Field Installation Errors: Technician installs meters at wrong addresses
  • Data Entry Mistakes: Meter IDs transposed or entered incorrectly in billing system
  • Address Confusion: Similar addresses cause meter assignment to wrong premise
  • System Migration Issues: Data mapping errors during system conversions
  • Communication Module Swaps: Incorrect AMCD ID assignments during replacements

1.3 Real-World Example

Scenario: Two Residential Homes

Correct Configuration (Should Be):

  • SDP 00123456 (123 Main St) ‚Üí Meter A
  • SDP 00123457 (125 Main St) ‚Üí Meter B

Crossed Configuration (What Actually Happened):

  • SDP 00123456 (123 Main St) ‚Üí Meter B ‚ùå
  • SDP 00123457 (125 Main St) ‚Üí Meter A ‚ùå

Result: Customers are being billed for each other's consumption!

2. Why This Procedure is Critical

2.1 The Temptation to Take Shortcuts

‚ö†Ô∏è Common Mistake:
LDCs might be tempted to resolve crossed meter scenarios by simply:
  1. Ending the wrong SDP to Meter relationships via synchronization
  2. Starting the correct SDP to Meter relationships via synchronization
  3. Resubmitting meter read data

WITHOUT requesting the removal of incorrect meter read data from MDM/R.

2.2 Why Shortcuts Fail

Skipping the Meter Read Data Removal step creates serious data anomalies:

Problem Impact
Message Sum Check Failures Old register reads remain in MDM/R and get mixed with new data, causing validation failures and unexpected estimations
Incorrect Estimations VEE system estimates based on register reads from different meters, producing wildly inaccurate values
Wrong Billing Quantities Customers receive bills based on mixed data from multiple meters, leading to billing disputes
Data Processing Failures Batch records contain mismatched register reads, causing processing exceptions
Additional Work Required Eventually you'll need to remove the data anyway, after investigating why billing is wrong

2.3 The Correct Approach

‚úì Follow This 4-Step Procedure:
Adherence to this procedure ensures that the accuracy of billing quantities is not compromised. The Meter Read Data Removal Request submitted to SME cleanly removes all incorrect data, preventing the data anomalies described above.

2.4 Timeline

Processing Time: A properly formatted Meter Read Data Removal Request is processed within 5 business days from the date it is submitted by the LDC or their Authorized Agent.

3.1 Process Overview

The crossed meter correction process is a multi-step effort involving:

  • Synchronizations to end and start relationships
  • Removal of incorrect meter read data via Service Desk request
  • Re-loading of correct meter read data

All steps are required to obtain correct billing quantities.

üß™ Practice First:
LDCs are encouraged to practice in the Sandbox environment to customize these steps according to internal system and process requirements before performing in production.

4. STEP A: End Incorrect SDP to Meter Relationships

STEP A Overview

Responsible: LDC

Purpose: End the existing incorrect SDP to Meter relationships via synchronization

Result: Relationships are terminated in MDM/R, preparing for data removal

4.1 Procedure

Step A.1: Generate Synchronization File Set

  • Follow your internal process to generate an incremental synchronization
  • End the existing SDP to Meter relationships (meter removal scenario)
  • Include all associated elements
CRITICAL RULE: Relationship End Date/Time
The End Date/Time of the SDP to Meter Relationship must be equal to or later than the Start Date/Time for each relationship.

If dates are equal, this is known as a "crush" (ending and starting on the same date/time).

Step A.2: Submit Synchronization

  • Submit the file set via FTS
  • Wait for MDM/R to process

Step A.3: Confirm Success

  • Check the IR06 Report to ensure each SDP to Meter Relationship has been updated
  • If SDP is NOT listed on IR06, check the IR07 Report for errors
  • Resolve any issues and repeat from Step A.1
  • When SDP is listed on IR06, confirm in the MDM/R GUI that each SDP to Meter Relationship has been end-dated

4.2 Example File Records

Situation: Ending incorrect relationship for SDP 00123456 with Meter B

‚ö†Ô∏è Remember: Do NOT end the Meter to Communication Module relationship when removing a meter (Business Rule 20).

5. STEP B: Remove Incorrect Meter Read Data

STEP B Overview

Responsible: LDC or Authorized Agent (submits request), SME (processes request)

Purpose: Permanently remove incorrect meter read data from MDM/R

File Type: 7900 (Request)

Processing Time: 5 business days

5.1 Two Methods for Creating Requests

You can create the Meter Read Data Removal Request in two ways:

Method Description
Method 1: Service Desk Generated Easiest option. Service Desk Tool automatically creates the required files. Maximum 10 SDPs per request.
Method 2: LDC/Agent Generated Manual creation following Meter Read Data Removal Request/Response TIS. Provides more control and flexibility.

5.2 Method 1: Service Desk Generated Files (Recommended)

Step B.1: Access Service Desk Tool

  • Log into the MDM/R Service Desk Tool
  • Create a new ticket: "MRD Removal Request - System Gen Files"

Step B.2: Enter Required Information

  • Select the environment (Production, UAT, Sandbox)
  • Enter Universal SDP IDs (maximum 10 per request)
  • Enter Meter IDs
  • Specify time range start date/time
  • If you're an Authorized Agent, specify the LDC
Time Range: Data from the specified TIME_RANGE_START to the current date/time (when SME processes the file) will be disassociated from the SDP.

Step B.3: Submit Request

  • Review all information for accuracy
  • Submit the request
  • Note the ticket number for tracking

The system automatically creates the required file and assigns it for processing. Proceed to Section 5.4 (Processing the Request).

5.3 Method 2: LDC/Agent Generated Files

Step B.1: Create Request File

  • Follow business rules and specifications in the Meter Read Data Removal Request/Response TIS
  • Use File Type 7900
  • Include all required fields

Step B.2: Generate MD5 Hash

  • Use an MD5 Message-Digest Algorithm tool
  • Generate a 128-bit (16-byte) hash value as specified in RFC 1321
  • Save the MD5 hash-code for inclusion in the Service Desk ticket
MD5 Purpose: The MD5 checksum validation ensures the integrity of submitted files and protects against file corruption during transfer.

Step B.3: Submit via Service Desk

  • Create a new ticket: "MRD Removal Request - LDC Gen Files"
  • Select the environment
  • If Authorized Agent, specify the LDC
  • Select number of SDP-Meter request records (max 10)
  • Include the MD5 Hash Code in the Description field
  • Attach the request file (only one file per ticket)
  • Submit

File Name Validation Requirements

The following requirements must be met before the ticket can be submitted:

Requirement Description
Organization Requesting organization must be an LDC or Authorized Agent
ORG_ID Matching (LDC) If LDC submits: ORG_ID of user, ORG_ID_1, and ORG_ID_2 in file name must be the same
ORG_ID Matching (Agent) If Agent submits: ORG_ID_2 must match Agent ORG_ID; Service Desk request must specify LDC ORG_ID
Unique File Name DATE_TIME element must be unique across all requests submitted by an LDC
ORG_ID in File ORG_ID_1 in file name must match LDC ORG_ID on each record inside the file
SDP Count Number of SDPs must be ‚â§ 10
Field Lengths Universal SDP ID (8 digits), SDP ID (50 chars), Meter ID (50 chars)

5.4 Processing the Request

What SME Does

  1. SME processes the Meter Read Data Removal according to specifications
  2. All specified data is permanently removed (cannot be undone)
  3. SME sends notification when request state is updated to "Resolved"
  4. Resolution Code is set to "Standard Service Request Completed"
‚ö†Ô∏è PERMANENT REMOVAL:
The data removal is permanent and cannot be undone by the SME. You MUST resubmit the correct meter read data after removal.

Output Files

The completed ticket includes three output files plus their MD5 validation files:

File Type File Name Example Contents
7901 ORG_ID_1.ORG_ID_2.7901.00.yyyyMMddHHmmss.00.DAT Register read data that was removed
7902 ORG_ID_1.ORG_ID_2.7902.00.yyyyMMddHHmmss.00.DAT Interval data that was removed
7903 ORG_ID_1.ORG_ID_2.7903.00.yyyyMMddHHmmss.00.DAT Error report (any errors encountered)

Step B.4: Validate Output Files

  • Download the three output files and their corresponding .MD5 files
  • Validate files for non-repudiation by checking MD5 hash-codes
  • If there's a mismatch, notify SME within 14 calendar days using the same ticket

Step B.5: Review Error Report

  • Review the error report output file (7903)
  • Review ticket notes
  • Refer to Meter Read Data Removal Request/Response TIS Section 2.11.3 for error descriptions

Step B.6: Handle Errors (If Any)

If errors are encountered:

  • Correct the error in your source data
  • Create a NEW meter read data removal request
  • Generate NEW request files (use unique DATE_TIME in file name)
  • Submit as a new ticket

Step B.7: Verify Data Removal

  • Access the MDM/R GUI
  • Search for the affected SDPs
  • Verify that the incorrect meter read data has been removed
  • Check the date range to ensure all specified data is gone

6. STEP C: Start Correct SDP to Meter Relationships

STEP C Overview

Responsible: LDC

Purpose: Start the correct SDP to Meter relationships via synchronization

Result: Meters are now properly associated with correct SDPs

6.1 Procedure

Step C.1: Generate Synchronization File Set

  • Follow your internal process to generate an incremental synchronization
  • Start the CORRECT SDP to Meter relationships (meter installation scenario)
  • Include all associated elements
Important Timing Rule:
If you did NOT submit a "crush" in Step A (i.e., the End Date/Time was NOT equal to the Start Date/Time), then the Start Date/Time of the new SDP to Meter Relationship must not overlap with a period where the meter was in-effect at any other SDP.

Step C.2: Submit Synchronization

  • Submit the file set via FTS
  • Wait for MDM/R to process

Step C.3: Confirm Success

  • Check the IR06 Report to ensure each SDP to Meter Relationship has been updated
  • If SDP is NOT listed on IR06, check the IR07 Report for errors
  • Resolve any issues and repeat from Step C.1
  • Confirm in the MDM/R GUI that the correct SDP to Meter Relationships are now Active

7. STEP D: Load Correct Meter Data

STEP D Overview

Responsible: LDC

Purpose: Resubmit the CORRECT meter read data for the affected SDPs

Result: MDM/R now has accurate data for billing quantity generation

7.1 Critical Timing Requirement

‚è∞ WAIT ONE DAY:
Load the meter read data the day after the synchronization (Step C) has been completed.

This ensures that no timing discrepancies occur between updating master data and all applicable tables in MDM/R and the processing of meter read data.

7.2 Procedure

Step D.1: Obtain Correct Meter Read Data

  • Retrieve interval data AND register reading data for each meter
  • Data must be for the CORRECT SDP to Meter relationships
  • Start from the Start Date/Time of the SDP to Meter Relationship
  • Ensure data matches the actual meter that should have been at the SDP

Step D.2: Wait for Sync Completion

  • Ensure Step C synchronization completed successfully (IR06 confirmation)
  • Wait until the next day (24 hours after sync completion)

Step D.3: Resubmit Meter Read Data

  • Submit meter read data files for each affected SDP
  • Follow standard Meter Read submission procedures
  • Include both interval data and register reads

Step D.4: Verify Data Loading

  • Access the MDM/R GUI
  • Search for the affected SDPs
  • Verify that meter read data has been loaded
  • Check date ranges to ensure all data is present
  • Verify VEE processing status

Step D.5: Request Billing Quantities

Once meter read data is loaded:

  • Submit an off-cycle Billing Quantity Request
  • Review the billing quantities for accuracy
  • Compare to expected consumption patterns
  • Verify no unexpected estimations occurred

8. What Happens If You Skip Data Removal?

This section explains the data anomalies that occur when LDCs attempt to resolve crossed meters WITHOUT requesting meter read data removal.

8.1 Scenario 1: Message Sum Check Failures

The Problem

What LDC Does (Incorrectly):

  • Ends/starts SDP to Meter relationships via sync
  • Resubmits ALL meter read data aligned with existing timestamps
  • Does NOT request data removal

Unexpected Result: Message Sum Check fails and unexpected estimations occur

Why This Happens

The MDM/R performs a sum check on Meter Read Blocks submitted for an SDP. If the absolute value of the sum check difference exceeds the VEE service threshold, the check fails and interval data is estimated.

Technical Explanation:
  1. When meter read data is submitted, there's no guarantee it's inserted in chronological order
  2. Old (incorrect) register reads remain in MDM/R
  3. System may use an old register read as the Start Register Read for Message Sum Check
  4. Mismatch between old and new register reads causes validation failure
  5. Estimations are triggered based on mixed register reads from different meters
  6. Billing quantities become inaccurate

Visual Example

Day 1 Day 2 Day 3 OLD DATA (Incorrect Meter B at SDP 1): Register: 100 -----> 150 -----> 200 NEW DATA (Correct Meter A at SDP 1): Register: 100 -----> 120 -----> 150 What Should Happen: Day 3 interval sum = 150 - 120 = 30 kWh ‚úì What Actually Happens (without data removal): System uses OLD register (150) + NEW register (150) Day 3 sum check: 150 - 150 = 0 kWh ‚ùå Result: Estimations triggered, billing quantities wrong!

8.2 Scenario 2: Wrong Billing Quantities

The Problem

What Happens: The presence of previously estimated register readings leads to calculation of register reading values that have NO relation to actual meter readings.

Example

Before Detecting Crossed Meters: SDP 1: Meter A data stored (actually should be Meter B) Day 1: 100, Day 2: EST, Day 3: 300 SDP 2: Meter B data stored (actually should be Meter A) Day 1: 100, Day 2: 150, Day 3: 200 After "Fixing" Without Data Removal: SDP 1: Has BOTH Meter A and Meter B register reads Meter A: 100, EST, 300 Meter B: 100, 120, 150 Result: Billing calculations use mixed register reads ‚Üí Completely incorrect billing quantities!

8.3 The Solution

‚úì Always Follow the 4-Step Procedure:
  1. End incorrect relationships (Step A)
  2. Request data removal from SME (Step B) ‚Üê Critical!
  3. Start correct relationships (Step C)
  4. Load correct meter data (Step D)

This cleanly removes ALL incorrect data and prevents the anomalies described above.

10. Quick Reference Checklist

10.1 Pre-Work

  • Crossed meter scenario identified and confirmed
  • Know which meters are crossed (Meter IDs)
  • Know which SDPs are affected (Universal SDP IDs)
  • Know date/time range of incorrect data
  • Have access to correct meter read data
  • Consider practicing in Sandbox environment first

10.2 STEP A Checklist

  • Generate synchronization to end incorrect relationships
  • End Date/Time ‚â• Start Date/Time
  • Do NOT end Meter to Comm Module relationships
  • Submit file set via FTS
  • Check IR06 for success
  • Verify in GUI that relationships are end-dated

10.3 STEP B Checklist

  • Choose method: Service Desk generated OR LDC generated
  • Create Meter Read Data Removal Request
  • Maximum 10 SDPs per request
  • If LDC generated: create MD5 hash
  • Submit via Service Desk ticket
  • Wait 5 business days for processing
  • Download and validate output files (7901/7902/7903)
  • Review error report (7903)
  • Verify data removal in GUI

10.4 STEP C Checklist

  • Generate synchronization to start correct relationships
  • Ensure no overlap with incorrect relationship period
  • Submit file set via FTS
  • Check IR06 for success
  • Verify in GUI that correct relationships are Active

10.5 STEP D Checklist

  • Wait one day after STEP C completion
  • Obtain correct meter read data for affected SDPs
  • Data from SDP to Meter relationship Start Date/Time
  • Include both interval data and register reads
  • Submit meter read files
  • Verify data loaded in GUI
  • Request off-cycle billing quantities
  • Review billing quantities for accuracy

10.6 Common Pitfalls to Avoid

‚ùå DO NOT:
  • Skip the Meter Read Data Removal request (Step B)
  • End Meter to Communication Module relationships
  • Submit Step D data before Step C is complete
  • Submit Step D data on the same day as Step C
  • Include more than 10 SDPs in a single removal request
  • Forget to verify data removal in GUI
  • Skip MD5 validation of output files
