1. Overview

This guide provides comprehensive instructions for synchronizing meter changes to the MDM/R system. A meter change involves removing an existing meter from an SDP and installing a new meter (or restarting the same meter).

What This Guide Covers:
  • Scenario A: Physical Meter Change - Same Configuration (Most Common - ~80% of cases)
  • Scenario B: Physical Meter Change - Different Configuration (Requires 2 syncs)
  • Scenario C: Logical Meter Change (Same physical meter, restart relationship)
  • CT/PT Multiplier changes (optional, can occur with any scenario)
Key Difference from New Installation:
Meter changes require ending existing relationships before or while starting new ones. The SDP already exists in MDM/R - you're only changing the meter configuration.

1.1 When to Use This Guide

Use this guide when:

  • Replacing a failed or damaged meter
  • Upgrading meter technology (e.g., older AMI to newer model)
  • Correcting meter installation errors
  • Performing meter accuracy testing (swap and swap back)
  • Changing meter configuration (interval length, channel config)
Not Covered in This Guide:
  • Brand new SDP with first meter (see New Meter Installation Guide)
  • Permanent meter removal with no replacement (see Meter Removal Guide)
  • Meter read data resubmission after changes (separate process)

2. Quick Decision Flowchart

Which Meter Change Scenario Do I Need?

START: I need to change a meter at an existing SDP
↓
QUESTION 1: Are you physically swapping the meter?
(Removing one physical meter and installing a different physical meter)
↓ YES → Continue | NO → Go to Scenario C (Logical Change)
↓
QUESTION 2: Does the NEW meter have:
• SAME interval length as old meter? AND
• SAME channel configuration set as old meter?
↓ YES → Scenario A (Same Configuration) | NO → Scenario B (Different Configuration)
↓
QUESTION 3: Is there a CT/PT Multiplier change?
↓ YES → Add Section 7 (CT/PT Changes) | NO → Proceed with your scenario as-is
RESULT: You now know which section to follow!

3. Pre-Requisites (All Scenarios)

3.1 Before You Begin

  • SDP already exists in MDM/R (Universal SDP ID assigned)
  • Old meter is currently synchronized and active in MDM/R
  • New meter has been physically installed (for physical changes)
  • New meter is communicating with AMCC
  • You know the old meter's current configuration (Interval Length, Channel Configuration Set, Meter ID, AMCD ID)
  • You know the new meter's configuration (for physical changes)
  • You know if CT/PT Multiplier is changing
  • You have identified which scenario applies (A, B, or C)

3.2 Information You'll Need

Information Type Details Required
Universal SDP ID 8-digit USDP ID (already assigned to this SDP)
Old Meter Details Meter ID, AMCD ID, Interval Length, Channel Config Set, Dials, Installation Date/Time
New Meter Details Meter ID, AMCD ID, Interval Length, Channel Config Set, Dials
Swap Date/Time Actual date/time of meter removal and installation
CT/PT Status Current value (if non-unity), new value (if changing)
Important - Same Day Swap Emphasis:
This guide assumes you are removing the old meter and installing the new meter on the same day. The synchronization should occur after both the removal and installation are complete.

4. Scenario A: Physical Meter Change - Same Configuration

Scenario A Definition:
  • Physically removing one meter and installing a different meter
  • New meter has SAME interval length as old meter
  • New meter has SAME channel configuration set as old meter
  • Can be completed in ONE synchronization
  • Most common scenario (~80% of meter changes)
Reference: Business Scenario 3 (Table 2.3.10-3, page 73)

4.1 What Happens in Scenario A

In a single synchronization, you will:

  1. End the SDP to Old Meter relationship
  2. Keep the Old Meter to Communication Module relationship (do NOT end it)
  3. Start the SDP to New Meter relationship
  4. Start the New Meter to Communication Module relationship
  5. Start Dials parameter for new meter
⚠ CRITICAL RULE: Never End Meter to Comm Module When Removing Meter

When ending a meter relationship, you MUST provide the Meter to Communication Module relationship with the current Start Date/Time and NO End Date/Time. This is a common mistake that violates Business Rule 20.

4.2 Elements Required for Scenario A

File Element Action Notes
Asset (01) Old Meter Provide With OLD configuration (same as when installed)
Asset (01) Old Comm Module Provide Old AMCD ID
Asset (01) SDP Provide Existing SDP (for context)
Asset (01) New Meter Provide With SAME configuration as old meter
Asset (01) New Comm Module Provide New AMCD ID
Parameter (04) Dials (New Meter) START Match new meter installation time
Relationship (05) SDP to Old Meter END-DATE Current Start + End Date/Time
Relationship (05) Old Meter to Comm Provide, leave END null Current Start, End = null (CRITICAL)
Relationship (05) SDP to New Meter START New meter installation time
Relationship (05) New Meter to Comm START Same as SDP to Meter start
Timing Tip for Scenario A:
The End Date/Time for old meter and Start Date/Time for new meter should reflect actual field events:
  • End Date/Time: When old meter was physically removed
  • Start Date/Time: When new meter was physically installed and energized

These times are typically minutes apart (more common for same-day swap)

5. Scenario B: Physical Meter Change - Different Configuration

Scenario B Definition:
  • Physically removing one meter and installing a different meter
  • New meter has DIFFERENT interval length and/or channel configuration set
  • MUST be completed in TWO separate synchronizations
  • Less common but more complex (~15% of cases)
Reference: Business Scenario 18 (Table 2.3.10-18, pages 80-81)
WHY TWO SYNCHRONIZATIONS ARE REQUIRED:
When interval length or channel configuration set changes, the MDM/R must:
  1. End all channel relationships for the old configuration
  2. Create new channel relationships for the new configuration

These operations cannot happen simultaneously. File 07 (Component SDP) contains channel-to-channel mappings that are configuration-specific and must be cleanly separated.

5.1 Two-Sync Process Overview

Sync Purpose Key Actions
Sync 1 Remove old meter with old configuration End SDP to Meter relationship; Provide old meter with OLD config
Wait Period Verify Sync 1 processed successfully Review IR14, IR06, IR07; Check GUI
Sync 2 Add new meter with new configuration Start SDP to Meter relationship; Provide new meter with NEW config

5.2 Synchronization 1 - Remove Old Meter

Before Sync 1

  • Old meter has different interval length OR channel config set than new meter
  • Know old meter's CURRENT configuration
  • Old meter removed

Files for Sync 1

  • File 00: Manifest
  • File 01: Asset (old meter with OLD configuration, old comm module)
  • File 02: Premise (header only)
  • File 03: Service Agreement (header only)
  • File 04: Parameter (header only, or end Dials if required)
  • File 05: Relationship (end SDP-to-Old-Meter, provide Old-Meter-to-Comm)
  • File 07: Component SDP (header only)

Critical Checks Sync 1

  • Old meter provided with OLD interval length (not new)
  • Old meter provided with OLD channel config set (not new)
  • SDP to Meter relationship ended
  • Meter to Comm Module NOT ended

Between Syncs

  • Wait for Sync 1 processing to complete
  • Review IR14, IR06, IR07 reports
  • Verify in GUI old meter relationship ended
  • New meter installed (if not already)

5.3 Synchronization 2 - Add New Meter

Files for Sync 2

  • File 00: Manifest
  • File 01: Asset (SDP, new meter with NEW configuration, new comm module)
  • File 02: Premise (header only)
  • File 03: Service Agreement (header only)
  • File 04: Parameter (new meter Dials, CT/PT if applicable)
  • File 05: Relationship (start SDP-to-New-Meter, start New-Meter-to-Comm)
  • File 07: Component SDP (header only)

Critical Checks Sync 2

  • New meter provided with NEW interval length
  • New meter provided with NEW channel config set
  • New meter Start = Meter to Comm Start = Dials Start
  • Start Date/Time on or after old meter End Date/Time
⚠ Important Timing Note:
Do NOT submit Sync 2 until Sync 1 has been processed successfully by MDM/R. Review all reports and verify the old meter relationship has been properly ended before proceeding.

6. Scenario C: Logical Meter Change

Scenario C Definition:
  • NOT a physical meter swap (same meter stays installed)
  • End and restart the SDP to Meter relationship for the SAME meter
  • Used for administrative corrections or logical resets
  • Can be completed in ONE synchronization
  • Least common scenario (~5% of cases)
Reference: Business Scenario 4 (Table 2.3.10-4, page 74)

6.1 When to Use Logical Meter Change

Use Scenario C when you need to:

  • Reset meter data collection after a communication issue
  • Correct a synchronization timing error
  • Mark a logical boundary in meter history for administrative purposes
  • Re-establish meter relationship after resolving data issues
Note: This is NOT used for physical meter swaps. The same physical meter remains installed at the SDP.

6.2 Elements Required

File Element Action Notes
Asset (01) Meter Provide Same meter, same configuration
Relationship (05) SDP to Meter (ending) END-DATE Current Start + End Date/Time
Relationship (05) Meter to Comm Module Provide, leave END null Current Start, End = null (unchanged)
Relationship (05) SDP to Meter (restarting) START New Start (typically same as or after End)

6.3 Example Records - Scenario C

<FTSFN>ORG12345.ORG12345.4000.00.20250116080000.LOGIC1.05.01.DAT</FTSFN> H|ORG12345|IncrementalSync|Relationship|20250116080000 // END the existing SDP to Meter relationship Relationship|00123456|SDP|MTR-001|METER|20230308140000|20250115120000 // KEEP Meter to Comm Module unchanged Relationship|MTR-001|METER|AMCD-001|COMMUNICATION MODULE|20230308140000| // RESTART the SDP to Meter relationship (same meter) Relationship|00123456|SDP|MTR-001|METER|20250115120000|

7. CT/PT Multiplier Changes

Important: CT/PT Multiplier changes can occur with ANY of the three meter change scenarios (A, B, or C). This section provides the additional steps required when CT/PT is changing.

7.1 CT/PT Multiplier Rules

Business Rules 26-28 (pages 67-68):
  • Only non-unity values (NOT 1 or 0) are processed by MDM/R
  • Non-unity CT/PT Multipliers MUST be explicitly ended
  • CT/PT changes CANNOT be future-dated
  • A CT/PT Multiplier change MUST be accompanied by a meter relationship change

7.2 What to Synchronize for CT/PT Changes

Scenario Current CT/PT New CT/PT What to Synchronize
Adding CT/PT Unity (none) Non-unity (e.g., 100) Start new CT/PT parameter
Removing CT/PT Non-unity (e.g., 100) Unity (none) End existing CT/PT parameter
Changing CT/PT Non-unity (e.g., 100) Different non-unity (e.g., 200) End old + Start new CT/PT parameters
No CT/PT Change Unity Unity Nothing (no CT/PT sync needed)

8. Critical Date/Time Rules

⚠ All Date/Times MUST Be in EST

All date/time values in synchronization files MUST be in Eastern Standard Time (EST), not Eastern Daylight Time (EDT). This applies year-round, even during daylight saving time periods.

8.1 Key Timing Requirements

Element Timing Rule
Old Meter End Time Should reflect actual meter removal time
New Meter Start Time Should reflect actual meter installation time
Dials Start Time Must equal new meter SDP-to-Meter start time
Meter-to-Comm Start Must equal new meter SDP-to-Meter start time
CT/PT Start/End Must align with meter relationship changes

9. Common Mistakes to Avoid

❌ Mistake 1: Ending Meter to Comm Module

Never end the Meter to Communication Module relationship when removing a meter. This violates Business Rule 20.

❌ Mistake 2: Wrong Configuration in Sync 1 (Scenario B)

In Scenario B Sync 1, provide the OLD meter with its OLD configuration (not the new configuration).

❌ Mistake 3: Submitting Sync 2 Before Sync 1 Completes

For Scenario B, always wait for Sync 1 to process successfully before submitting Sync 2.

❌ Mistake 4: Misaligned Relationship Start Times

New meter's SDP-to-Meter, Meter-to-Comm, and Dials must all have identical Start Date/Time.

❌ Mistake 5: Not Ending Non-Unity CT/PT

Non-unity CT/PT Multipliers MUST be explicitly ended. They don't automatically end with meter change.

10. Post-Synchronization

10.1 Verify Reports

  1. IR10: Check for file set completeness issues
  2. IR14: Review for validation errors (check this first)
  3. IR06: Check for business rule violations
  4. IR07: Review processing exceptions

10.2 Verify in MDM/R GUI

  • Search for SDP ID
  • Confirm old meter relationship ended
  • Confirm new meter relationship active
  • Verify all parameters correct

10.3 Verify Meter Reads

  • Confirm new meter reads being received
  • Verify VEE processing working
  • Check billing quantities generation

11. Quick Reference Checklists

11.1 Scenario A Checklist

Scenario A: Physical Meter Change - Same Configuration

Before Synchronization

  • Old meter removed, new meter installed (same day)
  • Both meters have SAME interval length
  • Both meters have SAME channel configuration set
  • New meter communicating with AMCC
  • Know exact removal and installation times

Critical Checks

  • Old meter SDP to Meter relationship has End Date/Time
  • Old meter Meter to Comm Module has NO End Date/Time
  • New meter SDP to Meter Start = Meter to Comm Start = Dials Start
  • All date/times in EST
  • New meter interval length and channel config set match old

11.2 Scenario B Checklist

Scenario B: Physical Meter Change - Different Configuration (2 Syncs)

Sync 1 Critical Checks

  • Old meter provided with OLD interval length (not new)
  • Old meter provided with OLD channel config set (not new)
  • SDP to Meter relationship ended
  • Meter to Comm Module NOT ended

Sync 2 Critical Checks

  • New meter provided with NEW interval length
  • New meter provided with NEW channel config set
  • New meter Start = Meter to Comm Start = Dials Start
  • Sync 1 completed successfully before submitting Sync 2

11.3 Scenario C Checklist

Scenario C: Logical Meter Change

Critical Checks

  • Same Meter ID in both end and restart records
  • End Date/Time on interval boundary
  • New Start Date/Time equals or immediately follows End Date/Time
  • Meter to Comm Module unchanged (no end date)
  • Configuration unchanged

12. Summary & Comparison

12.1 Scenario Summary Table

End Meter to Comm? No No No

Complexity Medium High Low

12.2 Key Takeaways

Most Important Points:
  1. Configuration Determines Syncs: If interval length or channel config set changes, you MUST use 2 syncs (Scenario B)
  2. Only End SDP to Meter: Never end Meter to Comm Module when removing meter
  3. Match Start Times: New meter's SDP to Meter, Meter to Comm, and Dials must have identical Start Date/Time
  4. CT/PT Explicit End: Non-unity CT/PT Multipliers must be explicitly ended when changing or removing
  5. Verify Before Proceeding: For Scenario B, wait for Sync 1 to complete successfully before submitting Sync 2
