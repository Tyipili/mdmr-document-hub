1. VEE SERVICE PARAMETER

Defines the VEE (Validation, Estimation, and Editing) rule set that applies to a Universal SDP ID
Attribute Value
TIS Reference Section 2.2.6.4, Table 2.2.6.4-1
Parameter Type SDP Parameter (date-effective)
Future Dating ‚úì ALLOWED (Table 2.2.6.3-1)
Mandatory YES - Required for MDM/R processing (Table 2.1.2)

When Changes Are Required (Business Triggers)

  • Change in customer class requiring different validation rules
  • Switch from residential to small general service (or vice versa)
  • New VEE services deployed by MDM/R
  • Change in LDC business rules for data validation/estimation
  • Correction of incorrectly assigned VEE service

Required Synchronization Files & Fields

File Field/Element Notes
File 04 VEE Service Parameter Universal SDP ID + VEE Service Code + Start Date/Time
End Current Provide current VEE with End Date/Time Only if changing (not for initial setup)
Start New Provide new VEE with Start Date/Time Can be future-dated

Common VEE Service Codes

  • RES01: Residential Standard VEE
  • SGS01: Small General Service VEE
  • LGS01: Large General Service VEE
  • IND01: Industrial VEE

Business Rules & Validation

‚ö† Key Validation Points

  • VEE Service code must be valid and active in MDM/R
  • Start Date/Time cannot be in the past if creating new parameter
  • When changing VEE Service, old service must be ended before or at the same time new service starts
  • Gap in VEE Service coverage may cause meter read processing failures

Step-by-Step Synchronization Process

Scenario: Changing VEE Service from RES01 to SGS01

  1. Identify Change Date: Determine when new VEE service should take effect
  2. End Current VEE: Provide current VEE Service (RES01) with End Date/Time
  3. Start New VEE: Provide new VEE Service (SGS01) with Start Date/Time (equal to or after end of old)
  4. Submit Synchronization: Include in File 04 of sync file set
  5. Verify: Check IR14 and IR06 reports for any errors

üí° Pro Tip: Future Dating VEE Changes

VEE Service is one of the few parameters that can be future-dated. This allows you to schedule customer class changes in advance, such as when a customer upgrades their service level on a specific future date.

2. CT/PT MULTIPLIER PARAMETER

Defines the multiplier applied to meter readings for current transformers (CT) and potential transformers (PT)
Attribute Value
TIS Reference Section 2.2.6.6, Table 2.2.6.6-1; Business Rules 26-28
Parameter Type SDP Parameter (date-effective)
Future Dating ‚úó NOT ALLOWED - Must align with meter change
Mandatory CONDITIONAL - Only when multiplier ‚â† 1

‚ö† CRITICAL RULES FOR CT/PT MULTIPLIER

  1. Unity (1) is Default: If no CT/PT parameter exists, MDM/R assumes multiplier = 1
  2. Non-Unity Must Be Explicit: Only multipliers ‚â† 1 need to be synchronized
  3. Must End Non-Unity: When removing or changing from non-unity, you MUST explicitly end it
  4. Accompanies Meter Change: CT/PT changes must occur with meter relationship changes (physical or logical)
  5. No Future Dating: CT/PT Multiplier cannot be future-dated

When Changes Are Required (Business Triggers)

  • Installing new CT or PT equipment
  • Replacing existing transformers with different ratios
  • Correcting incorrectly synchronized multiplier values
  • Changing from direct metering to CT/PT metering (or vice versa)
  • Meter change that involves transformer configuration changes

Common CT/PT Scenarios

Scenario Current State New State Action Required
New Installation (Direct) None Unity (1) Nothing - default
New Installation (CT/PT) None Non-Unity (e.g., 100) START CT/PT parameter
Remove CT/PT Non-Unity (100) Unity (1) END CT/PT parameter
Change CT/PT Value Non-Unity (100) Non-Unity (200) END old + START new
No Change Non-Unity (100) Non-Unity (100) Nothing - continues

Required Synchronization Files & Fields

File Field/Element Notes
File 04 CT/PT Multiplier Parameter Universal SDP ID + Multiplier Value + Start/End Date/Time
File 05 Meter Relationship Must include meter relationship change (physical or logical)

‚ö† Common Mistake: Forgetting to End Non-Unity CT/PT

When removing transformers or changing to direct metering, many users forget to explicitly end the existing non-unity CT/PT parameter. This causes MDM/R to continue applying the old multiplier to new meter reads, resulting in incorrect billing quantities.

Solution: Always provide the CT/PT parameter with an End Date/Time when transitioning to unity (direct metering).

Step-by-Step: Changing CT/PT Multiplier

Example: Changing from 100:1 to 200:1 during meter swap

  1. Perform Meter Change: Physical or logical meter change (see Meter Change Guide)
  2. End Old CT/PT: In File 04, provide CT/PT parameter with value 100 and End Date/Time matching meter removal
  3. Start New CT/PT: In File 04, provide CT/PT parameter with value 200 and Start Date/Time matching new meter installation
  4. Ensure Alignment: CT/PT End/Start times must align with meter relationship End/Start times
  5. Submit & Verify: Check reports for Business Rule 26-28 compliance

Business Rule References

  • Business Rule 26: CT/PT Multiplier must be ‚â• 1
  • Business Rule 27: Only non-unity values (‚â† 1 and ‚â† 0) are processed
  • Business Rule 28: CT/PT changes must accompany meter relationship changes

3. DISTRIBUTOR RATE CLASS (DRC) PARAMETER

Defines the rate class used for distributor-related charges and settlement purposes
Attribute Value
TIS Reference Section 2.2.6.4, Table 2.2.6.4-2; Business Rule 7
Parameter Type SDP Parameter (date-effective)
Future Dating ‚úì ALLOWED (with midnight rule)
Mandatory CONDITIONAL - Depends on settlement requirements

‚ö† CRITICAL: Midnight Rule for DRC

Business Rule 7: Distributor Rate Class changes must be synchronized with Start Date/Time at midnight EST (00:00:00) of the effective date.

Why? DRC impacts daily settlement calculations. Starting at any time other than midnight disrupts the daily settlement process and causes business rule exceptions.

Correct: 2025-01-15 00:00:00 (midnight EST)
Incorrect: 2025-01-15 14:30:00 (2:30 PM EST)

When Changes Are Required (Business Triggers)

  • Customer rate class change (e.g., residential to commercial)
  • Changes in distributor billing arrangements
  • Correction of incorrectly assigned rate class
  • New rate class implementation by LDC
  • Seasonal rate changes (if applicable)

Common DRC Codes (Examples)

DRC Code Description
RES Residential - Standard residential customers
GS<50 General Service under 50 kW demand
GS50-999 General Service 50 kW to 999 kW demand
GS1000-4999 General Service 1000 kW to 4999 kW demand
LARGE Large User over 5000 kW demand
STL Street Lighting

üí° Pro Tip: Planning DRC Changes

Since DRC can be future-dated and must start at midnight, you can schedule rate class changes well in advance. This is particularly useful for:

  • Beginning-of-month rate changes
  • Seasonal rate transitions
  • Coordinating with billing cycle changes

Step-by-Step: Changing DRC

Example: Customer upgrading from Residential to General Service

  1. Identify Effective Date: Determine the date when new rate class takes effect (e.g., February 1, 2025)
  2. Calculate Midnight EST: Set Start Date/Time to 2025-02-01 00:00:00 EST
  3. End Current DRC: Provide current DRC (RES) with End Date/Time = 2025-02-01 00:00:00 EST
  4. Start New DRC: Provide new DRC (GS<50) with Start Date/Time = 2025-02-01 00:00:00 EST
  5. Submit Synchronization: Can be submitted before effective date (future-dated)
  6. Verify: Check IR14 and IR06 reports, ensure no Business Rule 7 violations

4. COMMODITY RATE CLASS (CRR) PARAMETER

Defines the rate class for commodity-related charges, primarily for retail energy market participants
Attribute Value
TIS Reference Section 2.2.6.4, Table 2.2.6.4-3; Business Rule 8c
Parameter Type SDP Parameter (date-effective)
Future Dating ‚úì ALLOWED (with midnight rule)
Mandatory CONDITIONAL - Required for retail energy customers

‚ö† CRITICAL: Midnight Rule for CRR

Business Rule 8c: Like DRC, Commodity Rate Class changes must be synchronized with Start Date/Time at midnight EST (00:00:00) of the effective date.

This ensures proper alignment with commodity settlement and billing processes.

When CRR is Required

  • Customer signs retail energy contract (Retailer)
  • Customer opts for Spot Market Pricing
  • Customer switches between RPP options (TOU, Tiered, ULO)
  • Customer enrolled in microFIT or FIT programs

Common CRR Codes

CRR Code Description
100 RPP - Residential Time-of-Use (TOU)
101 RPP - Residential Tiered Pricing
102 RPP - Non-Residential Time-of-Use (General Service < 50kW)
103 Retailer - Fixed rate retail energy contract
104 Spot Market Pricing - Hourly Ontario Energy Price (HOEP)
106 Ultra-Low Overnight (ULO) - Residential TOU with overnight pricing
999 Other - Programs like microFIT, FIT not in standard list

CRR vs DRC: Understanding the Difference

DRC (Distributor Rate Class): Governs distributor-related charges (delivery, grid maintenance)

CRR (Commodity Rate Class): Governs commodity/energy charges (actual electricity consumption cost)

Key Point: Both can exist simultaneously. A customer might be residential DRC but have a retail contract CRR.

Step-by-Step: Changing CRR

Example: Customer switching from RPP TOU to Retail Contract

  1. Identify Contract Start Date: Retailer contract effective March 1, 2025
  2. Set Midnight EST: Start Date/Time = 2025-03-01 00:00:00 EST
  3. End RPP TOU (100): Provide CRR 100 with End Date/Time at midnight
  4. Start Retailer (103): Provide CRR 103 with Start Date/Time at midnight
  5. Update ESP Relationship: In File 05, end old ESP relationship and start new retailer relationship
  6. Submit & Verify: Ensure Business Rule 8c compliance

5. FRAMING STRUCTURE PARAMETER

Defines how interval data is aggregated into billing quantities (daily framing boundaries and structure)
Attribute Value
TIS Reference Section 2.2.5, Table 2.2.5-1
Parameter Type Service Agreement Element (date-effective)
Future Dating ‚úì ALLOWED (with midnight rule)
Mandatory YES - Required for billing quantity production

‚ö† CRITICAL: Midnight Rule for Framing Structure

Framing Structure changes MUST be synchronized with Start Date/Time at midnight EST (00:00:00).

Why? Framing Structure defines daily boundaries for billing quantity creation. Starting at any time other than midnight disrupts the framing process and prevents proper bill determinant generation.

This is the most common error in synchronization!

What is Framing Structure?

Framing Structure determines:

  • How raw interval data is aggregated into billing quantities
  • Daily framing boundaries (typically midnight to midnight)
  • On-peak, mid-peak, off-peak period definitions (for TOU)
  • Seasonal variations in pricing periods
  • Special day handling (holidays, weekends)

Common Framing Structures

Framing Structure Description / Use Case
RES_TOU_SUMMER Residential Time-of-Use pricing, summer season (May 1 - October 31)
RES_TOU_WINTER Residential Time-of-Use pricing, winter season (November 1 - April 30)
RES_TIERED Residential Tiered pricing (flat rate up to threshold, higher rate above)
GS_TOU General Service Time-of-Use pricing
GS_DEMAND General Service with demand charges (kW billing)
NETMETER_TOU Net metering customers with TOU pricing

‚ö† Framing Structure & Channel Config Compatibility

Not all Framing Structures are compatible with all Channel Configuration Sets!

Example: Net metering framing structures require channel config sets that support bidirectional metering (separate delivered/received channels).

Before synchronizing: Verify compatibility in TIS Section 2.3.10 Business Scenarios.

When Changes Are Required

  • Seasonal changes (summer to winter TOU pricing)
  • Customer switches rate plans (TOU to Tiered, or vice versa)
  • Net metering installation
  • New tariff rates implemented by LDC
  • Customer class change requiring different framing

Required Synchronization Files & Fields

File Field/Element Notes
File 03 Service Agreement - Framing Structure Universal SDP ID + Framing Structure Code + Start Date/Time (midnight EST!)
End Current Current Framing Structure with End Date/Time Midnight EST on last day of old structure
Start New New Framing Structure with Start Date/Time Midnight EST on first day of new structure

Step-by-Step: Changing Framing Structure

Example: Seasonal Change - Summer TOU to Winter TOU

  1. Identify Transition Date: November 1, 2025 (start of winter pricing)
  2. Calculate Midnight EST:
    • End Summer: 2025-11-01 00:00:00 EST
    • Start Winter: 2025-11-01 00:00:00 EST
  3. End Summer Framing: Provide RES_TOU_SUMMER with End = midnight Nov 1
  4. Start Winter Framing: Provide RES_TOU_WINTER with Start = midnight Nov 1
  5. Future Date Allowed: Can submit this synchronization weeks/months in advance
  6. Submit & Verify: Check that framing boundaries are correct in GUI

üí° Best Practice: Schedule Seasonal Changes Early

Many LDCs schedule their seasonal framing structure changes months in advance. This ensures:

  • No last-minute synchronization issues
  • Smooth transition at season boundaries
  • Time to validate in non-production environments
  • Alignment with billing cycle schedules

6. RELATED REPORTS REFERENCE

Understanding the synchronization report outputs for successful implementation

IR10 - Incomplete Synchronization File Set Report

Attribute Details
Purpose Informs LDC that synchronization file set was not accepted by MDM/R
When Generated File set transmission incomplete for more than allowed time; missing mandatory files; incorrect file names
TIS Reference Section 2.3.2.2; SME-SPEC-84

IR14 - Synchronization Staging Table Loader Exception Report

Attribute Details
Purpose Provides status and exceptions encountered during Staging Table Loader process (file validation before synchronization)
When Generated After file set accepted by FTS and before business rule validation
What to Check File format errors, invalid codes, data type mismatches, missing required fields
TIS Reference Section 2.3.2.3; SME-SPEC-85

‚ö† ALWAYS CHECK IR14 FIRST!

If IR14 contains errors, the synchronization will NOT proceed to business rule validation. You will not receive IR06 or IR07 reports until IR14 issues are resolved.

Common IR14 Errors for High Priority Parameters:

  • Invalid VEE Service code
  • Invalid DRC or CRR code
  • Invalid Framing Structure code
  • Non-midnight timestamp for DRC/CRR/Framing Structure
  • Date format errors (not in YYYYMMDDHHMMSS format)

IR06 - Business Rule Exception Report

Attribute Details
Purpose Lists synchronization submissions that violate MDM/R business rules
Key Rules for High Priority Business Rules 7 (DRC midnight), 8c (CRR midnight), 26-28 (CT/PT rules)
Action Required Review exceptions, correct data, resubmit synchronization
TIS Reference Section 2.3.2.5; SME-SPEC-86

IR07 - Synchronization Processing Exception Report

Attribute Details
Purpose Provides detailed information on exceptions during synchronization processing
When Generated After staging table loader completes successfully, during business rule processing
Relationship to IR06 IR07 provides additional context for exceptions listed in IR06
TIS Reference Section 2.3.2.4; SME-SPEC-82

QUICK TIPS FOR SUCCESS

  • Always submit complete synchronization file sets with all 7 mandatory files (00-07)
  • Use Business Scenarios tables in TIS (Section 2.3.10) for step-by-step guidance
  • Review IR14 report first for validation errors before checking IR07 for business rule exceptions
  • Test synchronization changes in non-production environment when available
  • Keep sequence numbers sequential - contact MDM/R Service Desk if reset needed
  • For CT/PT changes, always perform with meter change (physical or logical)
  • Remember midnight EST requirement for DRC, CRR, and Framing Structure changes
  • Verify Channel Configuration Set compatibility with Net Metering framing structures
  • Future-dated changes require BOTH current state (with end date) AND future state elements
  • Non-unity CT/PT multipliers must be explicitly ended - unity (1) is the default
