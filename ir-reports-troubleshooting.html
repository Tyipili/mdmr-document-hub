1. Understanding MDM/R Synchronization Reports

Report Purpose and Functionality

The MDM/R system generates five types of reports during the synchronization process. Each report serves a specific purpose at different stages of processing:

Report Purpose When Generated Available in GUI Indicates Success
IR17 File Transfer Services (FTS) processing failures After AS2 processing, before file validation No (FTS only) N/A - Always indicates failure
IR10 Incomplete or out-of-sequence file sets During file set compliance check No (FTS only) N/A - Always indicates failure
IR14 File content validation errors (format, data type, structure) During staging table loader process No (FTS only) Code 00 = Success
Code 99 = Failure
IR06 Successful synchronization updates After MDM/R processes file set content (successful updates) Yes Yes - Lists successful updates
IR07 Business rule violations and processing exceptions After MDM/R processes file set content (failed updates) Yes No - Lists exceptions/failures

Key Concept: Sequential Processing

The MDM/R synchronization process works in stages. If a file fails at an earlier stage (IR17, IR10, or IR14), the later stages will not execute, and you won't receive IR06/IR07 reports. This is why you must troubleshoot reports in order.

2. Report Order & Troubleshooting Workflow

The Synchronization Process Flow

1
File Transfer (FTS)
LDC submits File Type 4000 Incremental Synchronization file set via AS2
2
FTS Processing
File passes AS2, then FTS validation (file name syntax, authorization, destination directory)
Failure = IR17 Report Generated
3
File Set Compliance
Check for complete file set, correct sequence number, valid manifest file
Failure = IR10 Report Generated
4
File Content Validation
Staging Table Loader validates data format, field lengths, data types, dates
Failure = IR14 Report Generated
5
MDM/R Processing
System processes file set content and applies business rules
Success = IR06 Report | Failures = IR07 Report

Troubleshooting Decision Tree

Critical: Always Start Here

Did you receive an IR17? If YES, stop and fix FTS issues first. File never made it past file transfer.

Did you receive an IR10? If YES, stop and fix file set issues. Staging table loader won't run.

Did you receive an IR14 with errors? If YES, stop and fix validation issues. MDM/R processing won't run.

Only after IR14 code 00 (success) will you receive IR06 and/or IR07 reports.

Expected Report Sequence for Successful Synchronization

  • No IR17 received (file transfer successful)
  • No IR10 received (complete file set, correct sequence)
  • IR14 received with code 00 (validation passed)
  • IR06 received (lists all successful updates)
  • IR07 may or may not be received (only if business rule violations occurred)

IR17 FTS Processing Failure Report

Report Functionality

The IR17 report is generated when a file successfully passes AS2 processing but then fails one of the subsequent File Transfer Services (FTS) processing steps. This occurs before any synchronization validation begins.

Important Note

IR17 is not the same as a Message Disposition Notice (MDN). MDNs relate only to AS2 processing. IR17 means the file passed AS2 but failed FTS validation.

FTS Error Codes - Complete Reference

Code Error Message Description & Resolution
E115 reading original embedded file name failed Cause: Could not extract the file name embedded in the body of the file.
Check: File name must be enclosed in <FTSFN>, </FTSFN> tags or missing entirely.
Resolution: Verify file name is properly embedded in file body with correct XML tags.
E116 can not obtain original file name Cause: Could not obtain file name from transport headers.
Check: File name not included in transport header by sender, or lost during encryption/decryption.
Resolution: Check AS2 configuration to ensure file name is included in transport headers.
E120 non-standard file name <original file name> Cause: File name syntax validation failure.
Check: File name does not match technical interface specifications.
Resolution: Verify file naming convention: ORG_ID_1.ORG_ID_2.FILE_ID.REQ_VER.DATE_TIME.DAT
E130 error reading/validating authorized submissions file <authorization file name> Cause: FTS configuration error - authorization file cannot be accessed.
Check: Internal FTS configuration issue.
Resolution: Contact MDM/R Service Desk - this is a system configuration issue.
E140 submission of file <original file name> is not authorized Cause: Authority validation failed.
Check: Your organization is not authorized to submit this file type or for this LDC.
Resolution: Verify enrollment and agent relationships are properly configured with MDM/R.
E150 sender id <Sender ORGID> doesn't match org_id_2 from file name <original file name> Cause: AS2 sender ID doesn't match ORG_ID_2 from file name.
Check: Mismatch between AS2 configuration and file naming.
Resolution: Ensure ORG_ID_2 in file name matches your AS2 sender ID configuration.
E160 destination directory Cause: FTS configuration error - destination directory cannot be accessed.
Check: Internal FTS configuration issue.
Resolution: Contact MDM/R Service Desk - this is a system configuration issue.
E170 file <original file name> already exists Cause: Destination file already exists.
Check: File sent multiple times and already placed in destination directory.
Resolution: Check if file was already successfully submitted. If resubmitting, use different timestamp in file name.
E180 copying of <original file name> to <destination file name> failed Cause: FTS configuration error - file could not be copied to destination.
Check: Internal FTS configuration issue.
Resolution: Contact MDM/R Service Desk - this is a system configuration issue.

Troubleshooting Steps for IR17

  1. Identify the error code in the IR17 report
  2. For E115/E116/E120/E150: These are typically sender-side issues with file naming or AS2 configuration
  3. For E130/E140: These are authorization issues - contact Service Desk to verify enrollment
  4. For E160/E180: These are system-side issues - contact Service Desk immediately
  5. For E170: Check if previous submission was successful; adjust file name timestamp if resubmitting

IR10 Incomplete Synchronization File Set Report

Report Functionality

The IR10 report informs you that a synchronization file set was not accepted by MDM/R. This occurs during the file set compliance check, before content validation begins.

File Set Holding Period

Synchronization will hold and attempt to process incomplete or out-of-sequence file sets for up to one hour after receiving the files. If proper conditions aren't met within this threshold, files are cancelled and IR10 is generated.

Sequence Number Impact

When an IR10 report is received for a synchronization file set, the Sequence Number for the LDC will NOT be advanced. Synchronization only advances the expected Sequence Number upon successful processing where IR06 and IR07 reports are generated.

IR10 Condition Codes - Complete Reference

Code Condition Description Cause & Resolution
01 The synchronization file has been cancelled as the set of files has been incomplete beyond the allowable expiry threshold.

OR

No Manifest file submitted within the allowable expiry threshold.		 Scenario A: Files received (01, 02, 03, 04, 05, or 07) but no Manifest file within one hour.
Scenario B: Manifest file received but not all listed files received within one hour.

Resolution: Submit a complete synchronization file set with properly formatted Manifest file (File 00) and all required files (minimum: Files 01, 02, 03, 04, 05).
02 The synchronization file has been submitted out of sequence and will be held until the missing file set(s) have been submitted. Cause: Sequence Number in Manifest file doesn't match expected Sequence Number for the LDC. File set is being held (not cancelled).

This is a notification only - file will be processed when:
‚Ä¢ Missing file sets are submitted and processed first, OR
‚Ä¢ One hour expires (then receives Condition 01 or 03)

Resolution: Submit missing file sets with correct sequence numbers, or wait for timeout.
03 The synchronization file has been cancelled as it was submitted out of sequence and the missing file sets have been missing beyond the allowable threshold.

OR

Sequence Number is not of form Number (6).		 Cause A: No value provided for Sequence Number.
Cause B: Sequence Number in invalid format (must be exactly 6 digits, e.g., 000001, not 0001 or 00000001).
Cause C: Valid format but doesn't match expected Sequence Number after one hour.

Resolution: Check Manifest file header for Sequence Number format. Ensure it's 6 numeric digits and matches expected sequence. Submit complete file set with correct Sequence Number.
04 Malformed Manifest File

OR

One or more mandatory Files (01, 02, 03, 04, 05) are not listed in the Manifest file.

OR

File listed in Manifest is not a defined File No.		 Scenario A: Manifest file content improperly formatted (header or detail records).
Scenario B: Manifest doesn't list all mandatory files (must have 01, 02, 03, 04, 05).
Scenario C: File No. in Manifest is invalid (valid values: 01, 02, 03, 04, 05, 07).
Scenario D: Multiple IR10s generated - if one has Condition 04 while others have 01 or 03, ignore the Condition 04 (known defect).

Resolution: Review Manifest file structure against TIS specification. Ensure all mandatory files are listed with valid File No. values.

Understanding Sequence Numbers

Sequence Number Format

Valid format: Number(6) - exactly 6 numeric digits

Examples of valid Sequence Numbers: 000001, 000023, 123456, 999999

Examples of invalid Sequence Numbers: 0001 (too short), 00000001 (too long), 12AB56 (contains letters)

Rollover: After 999999, next expected Sequence Number is 000001

Troubleshooting Steps for IR10

  1. Check the Condition Code in the IR10 report
  2. Review "Sequence Number Received" vs "Sequence Number Expected" fields
  3. For Condition 01: Verify all files in set were submitted, especially Manifest file (00)
  4. For Condition 02: Check if this is a duplicate submission or if you skipped a sequence number
  5. For Condition 03: Validate Sequence Number format (must be exactly 6 digits) and value
  6. For Condition 04: Review Manifest file structure and ensure all mandatory files are listed
  7. Resubmit with corrections using the same Sequence Number (it wasn't advanced due to failure)

IR14 Staging Table Loader Exception Report

Report Functionality

The IR14 report provides exceptions for problems encountered during the staging table loading process for Incremental Synchronization file sets. This is the first content validation stage - it checks data formats, field lengths, data types, and structural integrity.

Critical: Check IR14 First

All exception error codes in IR14 (except code 00) will cause the synchronization file set to fail to load. IR06 and IR07 reports will NOT be generated if IR14 reports errors. Always resolve IR14 errors before expecting to see IR06/IR07.

Understanding IR14 Type Field

Type Meaning Description
F File Error Error at file level (e.g., header issues, file structure problems)
L Line/Field Error Error in a specific field of a detail record. The "Field" column will identify the problematic field.
R Record Count Summary of total records processed
C Field Count Summary of total fields processed
S Status Overall file set status (code 00 = success, code 99 = failure)

IR14 Error Codes - Complete Reference

Code Condition Description / Cause & Resolution
00 SUCCESS The file set was successfully loaded. This error code will only be populated if Type = "S".
Next Step: File set will proceed to MDM/R processing. Expect IR06 and/or IR07 reports.
01 ERROR File header is missing a parameter.
Check: Review file header record for all required parameters per TIS specification.
Resolution: Add missing parameter to header record.
02 ERROR File header is not consistent for file. LDC ID or Extracted Date Time values in header record are not the same for each file.
Check: Compare LDC_ID and Extracted_Date_Time across all files in the set.
Resolution: Ensure all files in set have identical LDC_ID and Extracted_Date_Time in headers.
03 ERROR File header indicates the wrong synchronization type.
Check: Verify synchronization type in header matches file content (e.g., Incremental vs Initial).
Resolution: Correct synchronization type indicator in file header.
04 ERROR Field does not conform to the required data format/length.
Check: Review the specific field identified in the "Field" column against TIS data format requirements.
Resolution: Correct field format/length to match specification (e.g., Char(X), Varchar(X), Number(X)).
05 ERROR Field contains invalid data. Data was not loaded into database. One or more field errors in a record.
Check: Field identified contains data that doesn't match allowed values or format.
Resolution: Review allowed values for the field and correct data content.
07 ERROR The Date/Time submitted is invalid (e.g., February 31, or 26:00:00).
OR Date/Time is out of the range configured in MDM/R as acceptable.
Check: Validate date/time values are real dates and within acceptable range.
Resolution: Correct invalid dates (check for impossible dates like Feb 31) or ensure dates are within system-configured acceptable range.
09 ERROR Header record does not match file number.
Check: File number in header doesn't match actual file type being submitted.
Resolution: Ensure file header correctly identifies the file number (01-07).
10 ERROR Universal SDP ID not defined through Universal SDP ID Assignment Process.
Check: The Universal SDP ID doesn't exist in MDM/R system.
Resolution: Verify Universal SDP ID exists through proper assignment process before synchronization.
11 ERROR Universal SDP ID does not match SDP and LDC ID.
Check: Mismatch between Universal SDP ID and the combination of SDP ID + LDC ID.
Resolution: Verify correct Universal SDP ID for the given SDP ID and LDC ID combination.
25 ERROR Incorrect number of fields in a file Header Record.
Check: Count of delimited fields in header record doesn't match specification.
Resolution: Verify header has correct number of pipe-delimited fields per TIS.
26 ERROR Incorrect number of fields in a file Detail Record.
Check: Count of delimited fields in detail record doesn't match specification.
Resolution: Verify detail record has correct number of pipe-delimited fields per TIS for that record type.
28 ERROR Start Date/Time is greater than the Extracted Date/Time of the file set. Future Date/Time not allowed for data element.
Check: Start Date/Time in record is later than file's Extracted Date/Time.
Resolution: Ensure Start Date/Time is not future-dated beyond Extracted Date/Time for this data element.
29 ERROR End Date/Time is greater than the Extracted Date/Time of the file set. Future Date/Time not allowed for data element.
Check: End Date/Time in record is later than file's Extracted Date/Time.
Resolution: Ensure End Date/Time is not future-dated beyond Extracted Date/Time for this data element.
30 ERROR Extracted Date/Time is future dated or older than 14 days from System Date.
Check: File's Extracted Date/Time is either in the future or more than 14 days old.
Resolution: Submit file set with current or recent Extracted Date/Time (within last 14 days).
31 ERROR One or more files do not contain a blank line at the end of file OR contain more than one blank line at end of file.
Check: End-of-file formatting issue.
Resolution: Ensure each file has exactly one blank line at the end, no more, no less.
33 ERROR Meter to Communication Module Relationship Start Date Time not found.
OR Meter to Communication Module Relationship not supplied for an SDP to Meter Relationship in the synchronization file set.
Check: Missing or mismatched Meter to Comm Module relationship.
Resolution: Ensure every SDP to Meter relationship has corresponding Meter to Comm Module relationship with matching Start Date/Time.
34 ERROR End Date is before Start Date.
Check: Logical date error - end date cannot be earlier than start date.
Resolution: Correct dates so End Date/Time is after Start Date/Time.
37 ERROR Duplicate Asset detail record found in the Asset Data File in the same Sync file set.
Check: Same asset appears more than once with same effective date.
Resolution: Remove duplicate records; ensure each asset record is unique within the file set.
40 ERROR Duplicate date-effective records in the same Synchronization file set.
Check: Same record appears with same effective date multiple times.
Resolution: Remove duplicates; consolidate into single record with correct dates.
42 ERROR Either or both Asset detail records not provided for a Meter to Communication Module Relationship.
Check: Relationship references assets that don't exist in the file set.
Resolution: Ensure both Meter and Comm Module asset records are present when defining relationship.
43 ERROR ORG ID supplied for an Agent is not defined in the MDM/R.
Check: Agent Organization ID doesn't exist or isn't enrolled in MDM/R.
Resolution: Verify Agent ORG ID is properly enrolled in MDM/R system before using in synchronization.
44 ERROR Provided value for Delivered USDP cannot reference itself.
Check: Universal SDP ID references itself as delivered USDP (circular reference).
Resolution: Correct Delivered USDP to reference different, valid Universal SDP ID.
99 FAILURE The file set was received and not loaded. This error code will only be populated if Type = "S".
Next Step: Review all error codes in IR14 report to identify and correct issues, then resubmit.

Common IR14 Error Patterns

Date/Time Related Errors (Codes 07, 28, 29, 30, 34)

  • Check for impossible dates (Feb 31, Apr 31, etc.)
  • Check for impossible times (25:00:00, 13:99:00, etc.)
  • Ensure Start Date/Time is before End Date/Time
  • Verify Extracted Date/Time is current (within last 14 days, not future)
  • Verify record dates are not future-dated beyond Extracted Date/Time when not allowed

Relationship Errors (Codes 33, 42)

  • Every SDP-to-Meter relationship requires corresponding Meter-to-Comm Module relationship
  • Start Date/Times must match exactly across related records
  • Asset records must exist before defining relationships
  • Never remove Meter-to-Comm Module when ending meter (Business Rule violation)

Troubleshooting Steps for IR14

  1. Check Type "S" record first - if code 00, file set passed validation; if code 99, file set failed
  2. Review all Type "F" and "L" records - these identify specific errors
  3. For Type "L" errors, note the "Field" column - tells you exactly which field has the problem
  4. Group errors by code - fix all instances of the same error type together
  5. Fix structural issues first (codes 01, 02, 03, 09, 25, 26) before data issues
  6. Validate dates comprehensively - date errors are very common
  7. Cross-check relationships - ensure all required relationships are present and valid
  8. Review Type "R" and "C" counts - helps identify scope of problems (how many records/fields affected)
  9. Resubmit corrected file set with same Sequence Number (wasn't advanced due to failure)

IR07 Synchronization Exception Report

Report Functionality

The IR07 report provides a complete listing of records that failed to update when submitted via Incremental Synchronization because a business rule violation or processing error occurred. This report is only generated after IR14 passes with code 00.

Key Differences: IR14 vs IR07

IR14 validates technical format, data types, structure (e.g., "Is this a valid date?")

IR07 validates business rules, data relationships, overlapping conditions (e.g., "Does creating this relationship violate business rules?")

Understanding IR07 Error Messages

IR07 error messages identify which table or relationship caused the failure. Use the table below to decode error messages and understand what to check:

IR07 Error Message Starts With Check These Items Common Issues
ACCT-SDP SDP-Account relationships Overlapping account relationships, missing account records, invalid effective dates
MTR-CHNL Meter-Channel relationships Invalid channel configuration, overlapping channel relationships, missing meter record
MTR-COMMOD Meter-Communication Module relationships Overlapping comm module relationships, missing asset records, date mismatches
MTR-PARAM Meter Parameters Invalid parameter values, overlapping parameter dates, unsupported parameter for meter type
SA Service Agreement records (Energy Purchase Services in MDM/R) Overlapping service agreements, invalid service type, missing prerequisite data
SDP-CTPT SDP-CT/PT parameters (SDP's CT/PT Device Multiplier in MDM/R) Overlapping CT/PT multipliers, attempting to change without meter change, invalid multiplier values
SDP-MTR SDP-Meter and Meter-Channel relationships Most common error - overlapping meter relationships, date mismatches between SDP-Meter and Meter-Channel
SDP-PARAM SDP Parameters Invalid parameter values, overlapping dates, midnight EST requirement violations (DRC, CRR, Framing)
SERVICE SDP's AMI Operator, Billing Agent, CCA Service Provider, and/or Energy Provider Service relationships Overlapping service relationships, invalid service provider IDs, missing service records

Interpreting Specific IR07 Error Messages

IR07 Error Message Contains What to Check & Resolution
...ChannelBO..MeterBO... Check: SDP-Meter and Meter-Channel relationships
Common Cause: Start Date/Time mismatch between SDP-Meter and Meter-Channel records
Resolution: Ensure SDP-Meter, Meter-Channel, and all related Dials have identical Start Date/Time
...MeterBO...SdpBO... Check: SDP-Meter and Meter-Channel relationships
Common Cause: Attempting to create overlapping relationships or relationship without required channel configuration
Resolution: Review Business Rules 18 and 20; ensure proper end-dating of previous relationships
Validation failed for "DeviceChannelRelBO...", An active record with no end date already exists Cause: SDP-Meter relationship violates synchronization rules - trying to create new relationship when active one exists
Resolution: End-date existing SDP-Meter relationship before creating new one (Business Scenario 18)
Validation failed for "DeviceChannelRelBO...", Start date of the record is before the end date of another record Cause: Overlapping SDP-Meter relationships - new relationship starts before previous one ends
Resolution: Adjust dates to ensure no overlap; gap or contiguous dates are OK, overlap is not
Validation failed for "DeviceFunctionRelBO...", An active record with no end date already exists Cause: Meter-Communication Module relationship violates rules - active relationship already exists
Resolution: Check Meter-Comm Module relationships; you cannot have multiple active relationships
Validation failed for "DeviceMultiplierBO...", An active record with no end date already exists Cause: Scaling Constant (SC) and Meter-Comm Module relationship violation
Resolution: Review SC Device Multipliers; end-date existing multiplier before creating new one
Validation failed for "SvcPtDeviceRelBO...", An active record with no end date already exists Cause 1: SDP-Meter relationship violates rules
Cause 2: SDP-CT/PT Parameter violates rules
Resolution: Check both SDP-Meter relationships AND SDP CT/PT Multipliers for overlapping active records. Remember: CT/PT changes must occur with meter changes.

Most Common IR07 Scenarios & Solutions

Scenario 1: Overlapping Relationships

Error: "An active record with no end date already exists"

Cause: Attempting to create new relationship without ending previous one

Solution: End-date the existing relationship in the same sync file set or in previous sync. New relationship can start at same time as old one ends (contiguous dates).

Scenario 2: Date Mismatches in Meter Changes

Error: Multiple errors referencing ChannelBO, MeterBO, or DeviceChannelRelBO

Cause: When installing new meter, SDP-Meter, Meter-Channel, and Dial Start Date/Times don't match exactly

Solution: All three must have identical Start Date/Time. Also ensure Meter-to-Comm Module relationship has same Start Date/Time.

Scenario 3: CT/PT Changes Without Meter Change

Error: Validation failed for SvcPtDeviceRelBO or SDP-CTPT errors

Cause: Attempting to change CT/PT multiplier in isolation

Solution: CT/PT multiplier changes must be performed with a meter change (physical or logical). Use 2-sync approach: Sync 1 removes old meter and CT/PT, Sync 2 adds new meter and CT/PT.

Scenario 4: Midnight EST Requirement Violations

Error: SDP-PARAM errors for DRC, CRR, or Framing Structure

Cause: Changes to Distributor Rate Class, Commodity Rate Class, or Framing Structure don't have midnight EST effective date

Solution: These parameters can only change at midnight EST (00:00:00). Adjust effective date/time to midnight.

Troubleshooting Steps for IR07

  1. Identify the record type causing the error (SDP, METER, SDP-MTR, etc.)
  2. Read the error message carefully - it often directly states the problem
  3. Check the MDM/R GUI to see current state of the entity (meter, SDP, relationships)
  4. Compare submitted data against MDM/R current state to identify conflicts
  5. Review relevant Business Rules in TIS Section 2.3.10 for detailed requirements
  6. For relationship errors, verify ALL related records are consistent (dates, IDs, statuses)
  7. Use Business Scenarios in TIS as templates for common transactions
  8. If multiple errors for same SDP, fix in logical order: Assets first, then relationships, then parameters
  9. Resubmit corrected records in new sync file set with next Sequence Number (IR07 means some records succeeded)

Note on Partial Success

IR07 indicates a partial success. Some records updated successfully (listed in IR06) while others failed (listed in IR07). You don't need to resubmit successful records - only correct and resubmit the failed ones.

IR06 Synchronization Updates Report

Report Functionality

The IR06 report provides a complete listing of records that were successfully updated via the Incremental Synchronization process. Only attributes that have been updated in the synchronization file set will be reported - unchanged attributes are not included.

This is a Success Report

Receiving an IR06 report means the synchronization file set passed all validation stages (FTS, file set compliance, staging table loader) and MDM/R successfully processed and applied the updates. This is the confirmation that your changes are now in the MDM/R system.

What IR06 Contains

The IR06 report includes the following record types, corresponding to the structure of synchronization file sets:

  • Synchronization Status: Overall summary of the sync file set processing
  • SDP Records: Service Delivery Point updates
  • Premise Records: Premise information updates
  • SDP Parameter Records: SDP parameter changes (VEE Service, CT/PT Multiplier, DRC, CRR, Framing Structure, etc.)
  • SDP to Account Relationship Records: Account relationship changes
  • Meter Records: Meter asset updates
  • Meter Parameter Records: Meter parameter changes (Interval Length, Channel Config Set, etc.)
  • SDP to Meter Relationship Records: Meter installation/removal records
  • Meter to Channel Relationship Records: Channel dial configurations
  • Comm Module Records: Communication module assets
  • Meter to Comm Module Relationship Records: Comm module associations
  • SDP to CTPT Asset Relationship Records: CT/PT asset relationships
  • SDP Service Records: Service provider relationship changes (AMI Operator, Billing Agent, Energy Provider, etc.)

Using IR06 for Verification

The IR06 report serves as your confirmation that changes were applied. Best practices for using IR06:

  1. Verify Sequence Number: Confirm the Sequence Number in IR06 matches your submission
  2. Cross-check updates: Review each record type to ensure all expected changes appear
  3. Compare with IR07: If you also received IR07, determine which records succeeded (IR06) vs failed (IR07)
  4. Document successful changes: Keep IR06 reports for audit trail and troubleshooting future issues
  5. Verify timing: Check Report Date/Time to confirm when changes were applied

IR06 + IR07 = Partial Success

You can receive both IR06 and IR07 for the same synchronization file set. This means:

  • IR06 lists records that successfully updated
  • IR07 lists records that failed to update
  • You only need to resubmit the failed records (from IR07) in your next sync

Report Delivery

Attribute Details
Delivery Method Sent via FTS to LDC and/or designated agents
GUI Access Yes - Available through Reports tab in MDM/R GUI
Timing Event-based - Generated upon completion of synchronization file set processing
File Format Pipe-delimited text file ("|" separator)

What to Do After Receiving IR06

Success Path

If you received ONLY IR06 (no IR07): Complete success! All records in your synchronization file set were processed and updated in MDM/R. No further action needed for this sync.

Partial Success Path

If you received BOTH IR06 and IR07: Partial success. Review IR07 to identify failed records, correct the issues, and submit a new synchronization file set with only the corrected records. Use the next Sequence Number.

Common Error Patterns & Quick Solutions

Symptom Likely Report Quick Solution
File never arrived at MDM/R IR17 or No Reports Check AS2 connection, file naming syntax, authorization setup. Verify MDN received successfully.
Missing files in set IR10 (Code 01) Submit complete file set with all mandatory files (00-05 at minimum) within one hour window.
Out of sequence IR10 (Code 02 or 03) Check expected Sequence Number in report. Submit missing file sets or wait for timeout, then resubmit with correct sequence.
Invalid date (Feb 31, 25:00:00) IR14 (Code 07) Review all dates for validity. Common error: copy/paste creating impossible dates.
Wrong field count IR14 (Code 25 or 26) Count pipe delimiters. Missing or extra delimiters are common. Review file structure per TIS.

CT/PT change fails IR07 CT/PT multiplier changes must occur with meter change. Use 2-sync approach: remove old (Sync 1), add new (Sync 2).
DRC/CRR/Framing change fails IR07 These parameters require midnight EST effective date. Change Start Date/Time to XX:00:00:00 (midnight).

When to Contact MDM/R Service Desk

Contact Service Desk for:

  • IR17 errors E130, E160, E180 (FTS configuration issues)
  • IR17 error E140 (authorization issues) if you believe you're properly enrolled
  • Persistent IR10 errors after confirming file set is complete and correctly sequenced
  • IR14 errors that don't make sense given your file content
  • IR07 errors that persist after following all business rules and scenarios
  • Any "null" error message in IR07 (internal system error)
  • Questions about Sequence Number resets
  • Suspected MDM/R system issues vs file issues

Service Desk: https://mdmrsupport.service-now.com
